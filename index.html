hexo theme

半个月前，在工作比较忙的情况下，我跟朋友说要用 pug(jade) 和 stylus 写一个 hexo 主题。我确实研究了一下，而且建了仓库写了一部分，过程中用了一个 generator-hexo-theme ，发现了一些问题也解决了一些，最终还是决定放弃。一方面是我在 hexo 官方的主题列表发现了目前使用的这个主题，我认为自己和这个或者说这类主题之间存着巨大鸿沟，与技术无关（实际上写一个静态博客能用到什么高深的技术呢），而是单纯的设计。我明白自己1、2年内都不可能写出比这么漂亮的主题了；另一方面，虽然可以勉强自称为伪全栈了，但我的兴趣完全不在前端，也根本就没想过走前端的路线，加上稍微研究了一下 hexo 的工作原理之后，我对开发一个属于自己的主题这件事也没那么大的兴趣了，毕竟写博客之前未必就要亲手写一个博客主题甚至引擎。

是有用 bootstrap 和 bottle 写过一个简洁的博客，我试着捡起来继续用它，但是过程中发现它是那么的蠢……狠抄了 Next 的前端，后端倒是挺简单的，借鉴 hexo 源文件的 Yaml + Markdown 毫无新意，最关键的是，并不是静态，而是存在了 Mongo 里，没有一个很好的 deploy 机制，都是我手动 rsync 的，虽然我知道做这些事也不过就是写一个脚本，deploy 之前比对一下就好了，可是它实在不是一个值得为其继续做出改动的东西。

另一个问题是，在被 DigitalOcean 和 ConoHa 分别坑了几个月之后，我又掉进了 vultr 的坑，要自己上传 arch 是挺坑的，然后手贱选了 coreOS ……对于一个对 docker 还停留在概念和玩过几下的人来说，用 coreOS 简直是噩梦。

想到我的 GitHub 是准备永生付费的2333，可不能浪费了，于是决定用 hexo serve 。

blog

已经好久没有写过博客了，也从没曾写出过一篇有意义的文章，我在半年前决定关闭博客的原因是：

大部分技术博客都是垃圾，内容不过只是一些排错和教程而已，而从质量上讲就更是垃圾了，去 stackoverflow 抄一记，搜来的东西就当是自己的解决问题的过程，有一些还不讲清楚环境和具体的触发原因甚至思路本身就是错的，误人子弟，文笔什么的……

也只有刚入行的程序员才觉得博客和 Geek 有关。至于《MacTalk》这本书（如果它也能算书）的作者和序作者，则是拉低技术圈博文质量的两位杰出代表。
